1. Situação atual (resumo)
Camada	O que temos
Interface	PWA (HTML/CSS/JS) em docs/ – 12 ficheiros principais
Lógica	8 scripts Python em scripts/ (scraping, verificação, OCR, notificações, estatísticas)
Automação	7 workflows GitHub Actions (crons, triggers, commits)
Dados	JSONs na cloud (apostas, sorteios, resultados, notificações)
Notificações	Web Push (VAPID + Service Worker)
OCR	API Gemini (chaves guardadas no GitHub Secrets)
Tudo isto tem cerca de 25 ficheiros de lógica + frontend. É uma app pequena.

2. Princípios da migração
Local-first: tudo corre no telemóvel, sem servidor.

Zero dependência do GitHub.

Notificações locais (sem FCM).

Base de dados leve: SQLite.

Scraping simplificado: se possível, usar HTTP requests diretos em vez de Selenium.

Framework sugerida: Flutter (multiplataforma, Dart é fácil de aprender, tem SQLite, WorkManager, notificações locais, câmara, etc.)

5. Desafios e soluções
Desafio	Solução
Scraping sem Selenium	Usar o endpoint não oficial que encontrei ao analisar o site da Santa Casa. Os dados vêm em JSON! (ex.: https://www.jogossantacasa.pt/web/ResultsBoard/EuroDreams – a página carrega dinamicamente, mas há chamadas XHR que podes reproduzir). Vou investigar para dar os URLs exatos. Se não, usar flutter_inappwebview para carregar a página e executar JS que extrai o conteúdo.
Execução em background	WorkManager no Android (com workmanager package) para agendar scraping diário. No iOS é mais limitado, mas a app pode verificar sempre que é aberta.
Armazenamento seguro das chaves	flutter_secure_storage para guardar as chaves Gemini.
Notificações locais	flutter_local_notifications para mostrar resultados quando a app corre em background (limitado) ou ao abrir.
Pré‑processamento de imagem	A biblioteca image do Dart permite redimensionar, converter para escala de cinzentos, etc. Pode‑se fazer uma versão simplificada.
